# Martin Training — Exercise Expansion & Swap Design

**Date:** 2026-03-22
**Status:** Approved

---

## Problem

1. The exercise pool per category/equipment tier is small (~4–6 exercises), causing the shuffle to repeatedly surface the same workouts.
2. The shuffle seed is date-based and deterministic — tapping shuffle on the same day always produces the same workout.
3. Users have no way to swap out an exercise they don't want without regenerating the entire workout.

---

## Goals

1. Expand the exercise library per category by researching exercises from popular fitness programs (GZCLP, PHUL, Reddit PPL, nSuns, 5/3/1, Starting Strength, Arnold Split).
2. Allow users to swap individual exercises inline without regenerating the whole workout.
3. Improve shuffle diversity by making manual reshuffles produce distinct results.

---

## Architecture

This is a single static HTML file using CDN React (no build step). All changes are made within `index.html`. The data format and component structure are preserved — no structural refactoring.

---

## Reference: dateSeed

`dateSeed(date)` is an existing function in the codebase:
```js
function dateSeed(d) {
  const dt = new Date(d);
  return dt.getFullYear() * 10000 + (dt.getMonth() + 1) * 100 + dt.getDate();
}
```
Output: a compact integer in YYYYMMDD format (e.g., `20260322`). Range: approximately 19700101–99991231. All seed arithmetic in this spec uses this as the base.

## Section 1: Exercise Pool Expansion

### Approach
Expand each category from ~4–6 to ~12–15 exercises per equipment tier (`full`, `home`, `minimal`). New entries follow the existing format:

```js
{ name: "Exercise Name", muscle: "Muscle Group", avoid: ["Area1", "Area2"] }
```

### Categories to expand
- `upper_push` — chest, shoulders, triceps
- `upper_pull` — back, biceps, rear delts
- `lower_push` — quads, glutes (push pattern)
- `lower_pull` — hamstrings, glutes (pull pattern)
- `core` — stability and anti-rotation

### Source programs
GZCLP, PHUL, Reddit PPL, nSuns, Wendler 5/3/1, Starting Strength, Arnold Split

### Scope note
In the existing code, `EX.crossfit = EX.hiit` and `EX.bands` is aliased to `EX.strength.minimal` across all categories. Only `EX.strength` and `EX.hiit` need direct expansion — crossfit and bands inherit the changes automatically.

### Example additions (full gym tier)
- **upper_push:** Close Grip Bench Press, Weighted Dip, Cable Lateral Raise, Pec Deck, Tricep Pushdown, JM Press
- **upper_pull:** T-Bar Row, Straight Arm Pulldown, Cable Lat Pullover, Hammer Curl, Preacher Curl, Chest-Supported Row
- **lower_push:** Leg Extension, Front Squat, Sissy Squat, Smith Machine Squat, Leg Press (Close Stance)
- **lower_pull:** Good Morning, Nordic Curl, Seated Leg Curl, Glute Ham Raise, Cable Kickback
- **core:** Cable Crunch, Decline Sit-Up, Ab Wheel Rollout, Windmill, Landmine Rotation

Home and minimal tiers receive equivalent dumbbell/band substitutions.

### Demo & video entries
All new exercises should have entries in both `DEMO` (form cues) and `VIDEO_MAP` (YouTube IDs). Missing entries are tolerated — the existing code already handles them gracefully (hides video element, skips form cues section). Exercises without entries are still valid pool members.

## Section 2: Exercise Swap UI

### Sets/reps/rest ownership
Sets, reps, and rest are **workout-slot properties** generated by `gen()` and attached to each exercise object in the generated workout array. They are not intrinsic to exercise definitions. Swapping replaces only `name`, `muscle`, and `avoid` on the slot — the slot's sets/reps/rest values are unchanged.

### Swap pool construction
The swap pool for a given slot is built at render time from the same category and equipment tier as that slot. Filtering is applied in this order:

1. Exclude the exercise currently displayed in this slot (by name).
2. Exclude exercises whose `avoid[]` intersects the user's active avoid list.
3. Exclude exercises currently displayed in **any other slot in the live workout** (post any prior swaps — read from current rendered state, not original generated list).
4. Equipment tier = user's selected tier exactly (`full`, `home`, or `minimal`) — no cross-tier fallback.

### Selection & cycling
- **State:** `swapCounts` — a flat object `{ [slotIndex]: number }`, initialised to `{}`.
- **Seed for slot `i`:** `dateSeed(date) + (i * 10000) + swapCounts[i]`
- **Collision analysis:**
  - Maximum workout slots: 5 (full-body day: upper_push + upper_pull + lower_push + lower_pull + core). So `i` ∈ {0,1,2,3,4}, and `i * 10000` ∈ {0, 10000, 20000, 30000, 40000}.
  - `shuffleCount` is expected to stay under 1000 in any session (human tapping limit).
  - `dateSeed` output ≈ 20,000,000. Slot seeds ≈ 20,040,000 max base. `swapCounts[i]` expected < 10000 per session. Total seed range stays well within JavaScript's safe integer limit (2^53 ≈ 9 × 10^15). No collision risk.
- **On tap:** Use null-safe increment — `const next = (swapCounts[i] ?? 0) + 1`. Update state: `setSwapCounts(prev => ({ ...prev, [i]: next }))`. Seed uses the post-increment value: `dateSeed(date) + (i * 10000) + next`. Then: `pick(filteredPool, 1, seededRandom(seed))[0]`.
  - **Why:** `swapCounts[i]` is `undefined` on first tap (starts as `{}`). `undefined++` produces `NaN` in JavaScript, which permanently corrupts the seed for that slot. The null-safe form ensures the seed is always a valid integer starting at `dateSeed + (i*10000) + 1` on the first tap.
- **Exhaustion / wrapping:** `swapCounts[i]` is never capped. The selected index is `next % filteredPool.length` (implicit in seeded pick). When `filteredPool.length === 1`, every tap returns the same exercise — this is correct and expected; the icon remains visible. When `filteredPool.length > 1`, taps cycle through alternatives.

### Icon visibility
- **Hidden** when `filteredPool.length === 0` (no valid alternatives after filtering).
- **Shown** otherwise, including when `filteredPool.length === 1`.
- Re-evaluated on every render — reactive to avoid list changes and workout mutations from prior swaps.
- There is no separate "no more options" state — if the pool has one entry, the icon shows and always returns that entry.

### State reset on reshuffle
When the user taps Shuffle (`shuffleCount` increments), `swapCounts` resets to `{}`. This ensures swap taps on a freshly generated workout start from a clean baseline.

## Section 3: Shuffle Diversity Fix

### Problem
`seededRandom(dateSeed(date))` is deterministic per day — tapping shuffle repeatedly returns the same workout.

### Fix
Add `shuffleCount` state (integer, starts at `0`). Seed becomes `dateSeed(date) + shuffleCount`.

- First load of the day: `shuffleCount = 0` → same stable daily default as before.
- Each shuffle tap: `shuffleCount++`, `swapCounts` resets to `{}`.

### Known limitations
1. **Page reload:** `shuffleCount` resets to `0`. The first shuffle of a new session on the same date produces the same workout as the very first shuffle of any prior session that day. Acceptable — date-based stability is the primary guarantee.
2. **Cross-date seed collision:** `dateSeed(day N) + shuffleCount=5` equals `dateSeed(day N+5) + shuffleCount=0` if `dateSeed` increments by 1 per day. This means, very rarely, two different date+shuffleCount combinations produce identical workouts. Acceptable — this is the same class of collision as the page-reload limitation and affects only edge cases.

## Error Handling

- **Empty shuffle pool:** Category pool entirely filtered by avoid list → fall back to `Bodyweight Circuit` (existing behavior unchanged).
- **Empty swap pool:** Hide the swap icon. Do not trigger a workout replacement.
- **`filteredPool.length === 1`:** Icon shows; every tap returns the same single alternative. No error state.
- **Missing DEMO entry:** Skip form cues section in demo modal (existing conditional rendering).
- **Missing VIDEO_MAP entry:** Hide video thumbnail area (existing `onError` logic).
- **Heavy avoidance + minimal tier:** Under maximum avoidance, swap icon may be hidden for all slots. This is correct behavior — no minimum pool guarantee is required.

## Testing

- All new exercises render in workout card and demo modal.
- Swap respects avoid list and equipment tier (exact match).
- Swap excludes exercises currently displayed in other slots (live state, not original generated list).
- Sequential swap taps on the same slot produce different exercises until pool is exhausted, then wrap.
- Slot i=4, shuffleCount=999: verify seeds do not collide (`20260322 + 40000 + N` vs `20260322 + 999`).
- Shuffle produces a different workout on each tap.
- `swapCounts` resets to `{}` after each shuffle.
- Swap icon hides when `filteredPool.length === 0`; re-appears if avoid list changes make alternatives available.
- `filteredPool.length === 1`: icon visible, tapping always returns same exercise, no crash.
- All body areas checked (max avoidance) — shuffle falls back to Bodyweight Circuit; swap icons hidden where applicable.
- Page reload on same date: first shuffle produces same workout as prior session's first shuffle (known limitation confirmed).
